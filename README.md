# Finance Agent（本地化财务报销 Agent 原型）

一个面向本地运行场景的财务报销 Agent 原型，当前重点聚焦差旅报销。  
核心不是“自由聊天”，而是“上传材料后如何被整理、核对、追问与修正”。

## 项目简介

本项目是一个本地化财务报销 Agent 原型，主场景为差旅报销处理。  
它既不是单纯 OCR 工具，也不是通用聊天机器人，而是采用“结构化意图解析 + 状态查询 + 轻量渲染”的方式，把用户追问落到可执行的任务状态上，提升上传材料后的处理效率与稳定性。

## 项目背景 / 解决的问题

传统报销处理常见问题是：材料类型混杂、缺件不透明、金额核对耗时、来回追问成本高。  
本项目要解决的是：用户上传材料后，系统如何持续帮助用户完成自动整理、缺失检查、金额核对和任务内追问处理，并在必要时支持轻量修正与状态同步。

## 核心功能

- 首页引导分流：根据用户目标和已上传材料，将会话路由到差旅/材料费/制度咨询流程。
- 差旅材料自动归类：对票据、支付记录、明细、酒店相关文件进行 doc_type 与 slot 归类。
- 缺失项检查：基于差旅槽位状态生成缺件清单。
- 金额不一致检查：对去程/返程/酒店关键金额进行比对并输出异常项。
- 任务内追问问答：对“还缺什么/哪些金额对不上/文件有哪些/下一步做什么”等追问做结构化解析并查询当前状态回答。
- 轻量修正与状态同步：支持在任务内进行重分类/槽位修正/金额修正等轻编辑动作并更新状态；支持导出任务结果（Excel / Text，以当前流程输出为准）。

## 系统流程

1. 用户上传票据或图片/PDF 材料。
2. 系统进行任务识别与差旅材料归类。
3. 构建当前任务状态（缺件、金额核对、槽位结果等）。
4. 用户在任务内继续追问（如缺什么、哪里对不上、下一步怎么做）。
5. 系统按结构化意图查询状态后回答，并支持轻量修正。
6. 任务确认后可导出结果文件（当前支持 Excel/Text）。

## 项目结构

以下为仓库中与主流程最相关的目录/文件：

- `app/`：核心业务代码
- `app/agents/`：多类 agent 角色与编排入口
- `app/usecases/`：业务用例层（含首页引导、差旅流程、travel_chat_service 等）
- `app/services/`：抽取、导出、RAG 等服务能力
- `app/db/`：数据库模型与仓储
- `benchmark/travel/`：差旅分类端到端评测说明与数据模板
- `data/`：运行期数据目录（数据库、上传文件、导出文件、索引）
- `scripts/`：评测/冒烟/数据集构建等脚本（如 `eval_travel_chat.py`、`travel_e2e_benchmark.py`）
- `test dataset/`：测试样例数据（按 go/return/hotel 细分）
- `requirements.txt`：Python 依赖列表
- `streamlit_app.py`：Streamlit 主入口

## 快速开始

### 1) 安装依赖

```bash
python -m venv .venv
# Windows
.venv\Scripts\activate
# macOS/Linux
# source .venv/bin/activate

pip install -r requirements.txt
```

### 2) 启动应用

```bash
streamlit run streamlit_app.py
```

### 3) 可选：运行回归/评测脚本

```bash
# 差旅追问链路轻量评估
python scripts/eval_travel_chat.py

# 差旅端到端分类评测（需准备 benchmark 数据集）
python scripts/travel_e2e_benchmark.py benchmark/travel/your_dataset.jsonl
```

### 4) 环境配置说明（保守）

项目会读取根目录 `.env`。如需启用本地模型或检索相关能力，请按本地环境配置对应变量（例如 Ollama / RAG 相关项）。  
若仅验证基础流程，可先按默认配置启动并逐步补齐环境项。

## 典型可交互问题

- 我现在还缺什么？
- 哪些金额对不上？
- 现在哪里对不上？
- 现在还有什么对不上？
- 酒店发票是哪几个文件？
- 酒店发票有几份？
- 我下一步该做什么？

- 返程呢？（承接上一轮金额问题）
- 这个文件是酒店发票。（轻量修正类输入）

## 技术方案与工程取舍

本项目没有把核心流程设计成完全开放式自由对话，而是优先保证“流程内可控性”：

- 使用结构化意图解析，将追问映射到有限业务意图（如缺件、文件列表、金额异常、下一步）。
- 使用状态驱动查询，只从当前任务状态（assignment/status）取事实，避免对关键事实“生成式瞎答”。
- 使用轻量文本渲染输出结果，保持回答自然但可追溯。

在个人项目约束下，这种方案能更好地平衡稳定性、算力成本和维护成本。  
仓库中也包含 LangGraph、RAG、Streamlit、本地模型调用等工程组件，用于支撑可扩展的实验与迭代。

## 当前限制

- 当前重点优化差旅报销流程，其他报销类型仍在扩展中。
- 对开放域闲聊不做重点支持，主要面向“任务内追问”。
- 复杂复合问句（多意图混合）目前以可控澄清为主。
- 当前版本定位为原型验证，工程能力持续迭代中。

## 后续规划

- 继续完善评测数据集与自动化回归脚本。
- 优化修正动作与导出链路的一致性与可观测性。
